# 架构说明

## 整体思路

这个项目按原生桌面应用来做，不走网页壳。界面分成两层：一个悬浮启动器，负责快速唤出助手；一个固定在右侧的侧边面板，负责承接诊断、修复和结果回看。诊断和动作执行都直接走本地模块，不再额外挂一层 Web 端服务。

```text
悬浮启动器
  -> 右侧侧边面板
  -> 本地诊断模块
  -> 问题分类和方案生成
  -> 邮件草稿导出
  -> 截图采集和导出
  -> 用户确认
  -> 本地修复动作执行器
  -> 结果回读、日志归档和附件落盘
```

## 运行分层

### 1. 启动层

- 悬浮按钮
- 全局或常驻入口
- 窗口固定与聚焦控制

### 2. 面板层

- 左侧或右侧侧边面板
- 当前场景展示
- 操作预览
- 结果回执

### 3. 诊断层

- 系统版本
- 根分区、内存、网络、音频、安装、打印状态
- 日志和设备枚举
- 当前窗口、剪贴板和定向截图

### 4. 分析层

- 问题分类
- 场景路由
- 处理建议
- 执行顺序
- 邮件草稿整理
- 截图和附件整理

### 5. 修复层

- 打印队列管理
- CUPS 过滤链修复
- 网络服务重启
- 音频会话重启
- 包管理修复
- 生成和保存动作运行日志

### 6. 交付层

- 支持包
- 诊断工单
- 动作日志
- 邮件草稿导出
- 截图导出
- 待执行脚本
- tarball 发布包

## 关键流程

1. 启动器被点开或唤起。
2. 侧边面板读取系统快照。
3. 助手判断当前问题落在哪个场景。
4. 先给出能看懂的方案，再让用户确认。
5. 需要修改系统时，动作先预览、再执行。
6. 执行后回读结果，留下日志、回执和导出文件。

## 设计取舍

- 先把最常见的桌面问题做稳，不追求一开始就覆盖所有场景。
- 本地诊断优先，关键动作交给本机命令和确认流程，不交给黑盒。
- 打印、网络、音频、安装这几类问题优先做闭环，因为它们最容易被真实用户碰到。
- 所有高风险动作都保留预览和确认，不做静默修改。
- 每次动作都要留下可回看的日志，方便后面补邮件、截图和问题说明。

## 打包方式

项目最终输出的是 `CPack` 生成的 `tar.gz`，方便在 deepin 上解包验证，也便于后续接成正式安装包。

打包后至少要能保留：

- 程序主体
- 诊断与动作模块
- 日志和回执目录
- 邮件草稿和截图导出目录
