# Orbit Deepin Assistant

一个面向 `deepin / UOS` 的系统智能助手前端原型，重点解决这几类场景：

- 系统常见问题排障
- 外设打印机连接
- 打印驱动配置 / 过滤链 / CUPS 修复
- 基于系统快照的感知问答

当前版本已经不是纯静态概念稿，除了未来科技感界面和故障树问答，还接入了本机实时诊断 API。

说明：

- 诊断 API 默认读取的是当前系统的运行状态和打印链路
- 不会把你工作区里的 `PPD` 或案例驱动文件当作默认输入
- 那些文件如果要用，只应该作为“某个厂商案例”的补充数据源

## 运行

要求：

- `Node.js 20+`
- `npm 10+`

启动开发环境：

```bash
cd /home/hao/AAA-PT2430PCz/deepin_assistant
npm install
npm run dev
```

这会同时启动：

- 前端：`http://127.0.0.1:4173`
- 本地诊断 API：`http://127.0.0.1:4174`

生产构建：

```bash
npm run build
```

单独启动本地诊断 API：

```bash
npm run start:api
```

## 当前能力

### 1. 沉浸式场景首页

- 以“地球 / 轨道 / 对接舱”作为主视觉
- 鼠标悬停模块时，场景重点和状态面板联动变化
- 已内置 4 个模块：
  - `System Orbit`
  - `Printer Link`
  - `Driver Forge`
  - `Perception Deck`

### 2. 感知问答控制台

- 可选择系统快照：
  - 发行版
  - 设备类型
  - 连接方式
  - 故障症状
- 根据问题文本匹配故障树分支
- 输出：
  - 健康评分
  - 风险标签
  - 故障阶段路径
  - 修复行动流
  - 命令清单

### 3. 本机实时诊断

当前已经接入本地 API，前端可以直接采集这台机器的真实数据：

- `/etc/os-release`
- `lpstat -t`
- `lpinfo -v`
- `lsusb`
- `journalctl -u cups`
- `systemctl is-active cups`

采集后会自动：

- 读取系统版本与内核
- 判断 CUPS 是否在线
- 分析当前打印队列状态
- 推断设备连接方式与故障症状
- 把结果映射回前端场景和建议命令

### 4. 执行控制台

当前已经接入动作层：

- `收集支持包`
- `导出诊断工单`
- `清空打印队列`
- `重启 CUPS 服务`

这些动作分为两类：

- 安全动作：只读导出，不修改系统状态
- 修复动作：会影响打印队列或系统服务，执行前会先给预览

当前动作执行结果已经包含：

- 执行时间线
- 动作日志文件
- 回滚 / 补救建议
- 最近动作历史

现在还新增了权限感知执行链：

- 动作卡片会显示当前环境的授权状态
- 系统级动作会区分：
  - `API 直接执行`
  - `需要人工授权`
  - `只能导出待执行脚本`
- 当 API 不能直接提权时，会自动生成：
  - 待执行 shell 脚本
  - `pkexec` / `sudo` 启动命令
  - 授权后的后续操作建议
- 手工授权脚本执行后，还会生成：
  - 授权回执 JSON
  - 授权执行日志
  - 前端轮询后的自动状态回填

### 5. 打印机 / 驱动问题演示

当前已覆盖的典型问题：

- USB / 网络打印机无法识别
- 打印队列卡住
- 驱动配置损坏 / 驱动不匹配
- CUPS 过滤链异常
- 标签纸宽 / 输出异常

## 当前技术结构

### 前端

- `React + Vite`
- 主场景模块联动
- 问答结果面板
- 实时诊断展示面板

### 本地服务

- `Node.js + child_process`
- 通过 `server/index.mjs` 调用本机系统命令
- 暴露接口：
  - `GET /api/health`
  - `GET /api/diagnostics/summary`
  - `GET /api/actions`
  - `GET /api/manual-actions/:id`
  - `POST /api/actions/:id`

## 下一步怎么继续做

### 1. 接打印修复执行器

当前已经有基础执行层，下一步适合继续拆成真正的“修复 Agent”：

- 重启 `cups`
- 清空旧作业
- 删除旧队列
- 重建打印机队列
- 重装驱动包
- 校验驱动配置和纸宽参数

建议把执行动作做成：

- 预览命令
- 用户确认
- 执行日志流
- 成功 / 失败回滚建议

### 2. 接文心或本地模型

当前问答是规则驱动，后续可以把 `createResponse()` 替换成模型路由层：

- 输入：
  - 用户问题
  - 系统快照
  - CUPS / 设备诊断结果
- 输出：
  - 故障解释
  - 修复步骤
  - 可执行命令
  - 风险提醒

推荐结构：

- `frontend`: 负责场景、状态和交互
- `orchestrator`: 汇总系统快照和工具输出
- `llm-router`: 调用文心 / 本地模型
- `executor`: 真正执行修复动作

### 3. 继续增强高权限动作

当前已经完成第二版高权限执行链，但还可以继续往下做：

- 区分 `polkit agent 不可用` 和 `sudo 权限不足`
- 把高权限动作扩展到：
  - 重建打印机队列
  - 删除旧队列
  - 重装关键驱动包
  - 修复 CUPS 过滤链权限

## 适合继续做的方向

如果继续做成参赛项目，下一阶段最值当的是：
